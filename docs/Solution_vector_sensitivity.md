# Solution vector sensitivity

This operational note explains why small changes in user need wording can have large effects on product, service, policy and outcome work.

Use this note when drafting, reviewing or changing user needs, especially when a need may influence solution direction, opportunity framing, outcome strategy or public-service accountability.

## Core idea

A user need is not only a sentence. It is a problem-space framing.

Small wording changes can shift the implied solution vector.

That means a team can pursue a desirable outcome while still solving the wrong problem if the need is framed at the wrong level, for the wrong actor, or with the wrong verb.

## Example: options and decision support

These needs are not equivalent.

| Need wording | Likely solution vector |
|---|---|
| Need to know options | Information provision, webpage, list, directory |
| Need to compare options | Comparison tools, criteria, filters, decision aids |
| Need to judge which options are realistic | Interpretation, advice, brokerage, contextual suitability |
| Need to decide which route to take | Decision support, confidence, consequences, timing, support network |
| Need to protect future options | Strategic planning, fallback routes, rights, sequencing, risk mitigation |

A one-word change can move the work from content, to comparison, to interpretation, to relational support, to rights and risk management.

## Why this matters for outcomes

Outcomes are not only affected by delivery quality.

They are affected by the maturity, precision and level of the problem framing.

If a project is trying to improve an outcome but the need is framed too narrowly, the work may produce outputs without changing the conditions that block the outcome.

For example:

```text
Need to know what options exist
```

may lead to a directory or guidance page.

But if the evidence shows the deeper need is:

```text
Need to judge which options are realistic for my circumstances
```

then a directory may not create the intended outcome. The real need may involve suitability, interpretation, support, timing, confidence, power and consequences.

## Operational rule

When user need wording changes meaning, treat it as a material change.

Record the change in the object changelog if the change affects:

- the actor
- the verb
- the need level
- the evidence basis
- the implied solution vector
- the related pain points or blockers
- the civic or capability implications
- the outcome strategy

Do not silently rewrite needs to make them clearer if the clarification changes what the team is likely to build, change, challenge or evaluate.

## Review prompts

When reviewing a user need, ask:

- What solution vector does this wording imply?
- Would a different verb imply a different response?
- Is this really an information need, a comparison need, an interpretation need, a decision need or a rights/support need?
- Is the actor correct?
- Is the need at the right level?
- Does the evidence support this level and wording?
- Would the stated outcome still be plausible if we solved only this need as currently worded?
- What pain points would remain unresolved?
- What civic needs or capability needs are implicated?

## Red flags

Review wording carefully when a need uses:

- `know`
- `understand`
- `be aware of`
- `access information`
- `make an informed choice`
- `navigate the system`
- `be independent`
- `be resilient`

These may be valid, but they can hide deeper needs around suitability, interpretation, agency, support, power, rights, trust or institutional responsibility.

## Relationship to LLM-assisted work

LLMs may make user needs sound clearer while changing the solution vector.

When using an LLM to draft or refine needs, check whether the wording has moved between different kinds of need, such as:

```text
know -> compare
compare -> judge suitability
understand -> decide
access -> secure support
choice -> protect future options
```

If the meaning has changed, preserve the original as part of the reasoning trail and record the change where appropriate.

## Relationship to outcome work

Outcome work should not treat the intervention as the only learning unit.

The evolving understanding of the need is also a learning unit because it can change the solution trajectory and the likelihood of achieving the outcome.

A useful outcome review should ask:

- Did the project address the need as originally framed?
- Did the understanding of the need evolve?
- Did the intervention change when the need changed?
- Did evidence show that the team was solving the wrong level of need?
- Did any policy, service or delivery constraint prevent the outcome despite the need being understood?
